GPOS–1: Pair Adjustment Positioning

The font for this test case contains a ‘GPOS’ table whose ‘kern’ feature enables a lookup of type 2 for Pair Adjustment Positioning. The lookup contains two subtables:
  • First, a PairPos format 1 subtable that adjusts the spacing between pairs of individual letters. For example, when Ą and J stand next to each other, their shapes would overlap. From a typographic perspective, this would look rather terrible, so this subtable puts them further apart.
  • Second, a PairPos format 2 subtable that adjusts the spacing betwen glyph classes. For example, when they appear on the left-hand side of a letter pair, this subtable categorizes the letters A Á Ą into one particular class, G Ģ into another, and V into yet another class. The same happens for the right-hand side: a á ą are put into one glyph class, f ﬁ ﬂ into another, and so on. Finally, the subtable contains a table how to change the spacing for each combination of classes. Compared to format 1, this encoding can be more compact.
If all the pairs in the rendering are badly spaced (eg. touching or overlapping), it’s a sign that your OpenType implementation does not handle ‘GPOS’ tables at all, or that it fails to process ‘GPOS’ tables with type 2 lookups.
If only some of the pairs are badly spaced, it could be a sign of a different problem. When an OpenType implementation processes a lookup, it should walk its subtables in the same order as listed in the font file. When a subtable matches, the implementation must use it to reposition the current glyph pair, and then it should continue with the next lookup. In other words, if the current lookup contains additional subtables beyond the one that matched, they should be ignored. In case your implementation fails to break its inner loop, thus continuing with the next subtable instead of the next lookup, it will wrongly position any glyph pairs that appear in multiple subtables of the same lookup. For example, in our test font, the pair Ąģ appears in both subtables of lookup #0. A correct OpenType implementation should only use the positioning from the first subtable.
