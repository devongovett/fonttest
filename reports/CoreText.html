The font for this test case is a variation font with a single axis. While the ‘gvar’ table does a regular interpolation without any special quirks, the font’s ‘avar’ table defines a mapping that modifies the variation axis.
A conformant OpenType implementation should produce the test glyph in its maximally thin shape at axis value 100. The resulting glyph should become bolder until 250, then stay at the exact same weight until 650, then become bolder up to the maximum at 900.

CMAP–1: Ideographic Variation Sequences

The Unicode Ideographic Variation Database defines glyph variants for Han ideographs. For example, Unicode supports two slightly different variants when rendering U+82A6 in Japanese; the font for this test case contains a type 14 ‘cmap’ table that implements them. With a conforming OpenType implementation, you should see four glyphs below. In the third glyph (U+82A6 U+E0101), the top stroke below the radical should be attached and slanting up. But in the first, second, and fourth glyph, that stroke should be detached and horizontal. If all four glyphs look the same, or if you see any black boxes in the renderings, it’s a sign that your OpenType implementation is broken.

CMAP–2: Unicode Variation Selectors

Unicode defines Standardized Variants of certain characters, including some Mathematical Symbols. CMAP–3: MacOS Turkish Encoding

The font for this test case contains a ‘cmap’ table of format 0, platform 1, encoding 0, language 18. Platform 1 means the classic Macintosh up to MacOS 9; the combination of encoding 0 and language 18 indicates the MacOS Turkish character encoding. Although Apple has deprecated the Macintosh encodings in favor of Unicode in 2002, present-day renderers may still encounter such fonts. For example, users sometimes want to open electronic documents that were created a long time ago, and certain file formats contain embedded fonts.
A correct OpenType implementation should be able to render Turkish text with this test font, using Unicode’s conversion table. If you see boxes, missing characters, or glyphs from a different (fallback) font in the rendering below, it’s a sign that your OpenType implementation cannot handle legacy fonts that were built for MacOS.

GVAR–1: Sharing All Points

The font for this test case uses shared point numbers in its ‘gvar’ table. In its shared points set, the font sets the number of points to zero. A correct OpenType implementation should interpret such an empty set as a compact encoding of “all points.” If the glyphs in the rendering below have all the same weight, of if some glyphs are missing entirely, it’s a sign that your OpenType implementation is broken.

KERN–1

The font for this test case contains a ‘kern’ table of format 0, which in turns contains a single format 0 subtable. Without kerning, the glyphs for T, u and the dotless ı will look like normal letters. However, if the OpenType implementation correctly applies the font’s kerning data, the shapes will join to form a geometric pattern. If you can still discern the original letters ıTuTuTı in the rendering, it’s a sign that the implementation fails to correctly handle kerning.
